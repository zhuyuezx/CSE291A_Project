\begin{abstract}
We present a self-evolving agent architecture where a large language model (LLM) improves search algorithms by \emph{writing and iteratively refining} executable heuristic tools offline, rather than making costly per-step decisions online. This framework introduces a novel decoupling: slow, reasoning-intensive LLM analysis is separated from fast, execution-heavy planning cores like Monte Carlo Tree Search (MCTS) and A*. During gameplay, the agent relies solely on the planning core equipped with pluggable Python heuristics. Post-gameplay, an offline continuous learning loop leverages LLM-driven trace analysis, task proposal, and code generation to autonomously evolve these heuristics based on failure traces. This architecture eliminates runtime LLM latency bottlenecks and yields transparent, auditable code artifacts instead of opaque neural weights. As detailed in our experiments (Section~\ref{sec:experiments}), this approach demonstrates strong generalizability across diverse domains. Evaluated on Sokoban, TextWorld, and Zork, our evolved heuristics achieved up to 74.4\% reductions in search node expansions and substantially improved zero-shot success rates on complex, unseen test instances. Code is available at \url{https://github.com/zhuyuezx/CSE291A_Project}.
\end{abstract}


\section{Introduction}
\label{sec:intro}

Solving turn-based games as Markov decision processes (MDPs) has proven to be an effective strategy, exemplified by successes in games such as Go. Monte Carlo Tree Search (MCTS) in particular searches the game environment and identifies promising paths algorithmically. AlphaZero-style approaches combine MCTS with deep neural networks for state value estimation, but they are difficult to set up and incur substantial entry cost. Meanwhile, prior work that integrates MCTS with large language models (LLMs) has largely focused on online, step-wise LLM guidance, which is costly and constrained by static, local views of the game state. The emergence of LLM-based tool creation and skill frameworks motivates an alternative direction: improving the policy by autonomously proposing and refining rule-based heuristics by LLM that the search algorithm can use.

We develop a self-evolving game agent where an LLM acts as an offline tool-creation agent. The system interacts with diverse environments (Sokoban, TextWorld, Zork), collects traces, analyzes feedback, and generates pluggable heuristics that are validated and iteratively improved. We use MCTS as the main framework. The LLM produces human-understandable code for key search phases and improvement appears even with few iterations. The LLM runs offline; heuristics run online effectively. This offers a cheaper path than per-step LLM calls or neural training, and yields interpretable code rather than weights.

To operationalize this offline tool-creation perspective and quantify its benefits, we make the following contributions toward robust, interpretable, low-cost planning:
\begin{itemize}
    \item We propose a self-evolving heuristic framework in which an LLM improves planning algorithms by generating executable, pluggable heuristic tools.
    \item We design an offline tool-evolution loop that analyzes gameplay traces and iteratively improves search policies without any runtime LLM calls.
    \item We demonstrate the framework across multiple domains including Sokoban, TextWorld, and Zork, achieving substantial gains in search efficiency and generalization.
\end{itemize}

\section{Related Work}
\label{sec:related}

\textbf{Monte Carlo Tree Search.} MCTS balances Monte Carlo exploration and exploitation in large decision spaces, often yielding strong decisions from limited rollouts\cite{Part2MCSurvey,Part2MCTSCoulom} and has been successful in combinatorial games and policy optimization\cite{Part2MCSurvey,Part2GoNNMCTS,Part2PGTS}, with continued improvements such as PUCT\cite{Part2MCSurvey}. AlphaGo-style systems augment MCTS with learned value/policy models\cite{Part2GoNNMCTS}; more recent work also integrates LLMs with MCTS, either to improve LLM reasoning or to use LLMs as search components/heuristic optimizers\cite{Part2PGTS,Part2MCLLM,Part2PlanHeuris}. We build on this line by keeping an MCTS backbone and using the LLM to generate programmable heuristics that optimize search behavior\cite{Part2PlanHeuris}.

\textbf{Sequential Decision-Making Games.} Games such as Zork, Go, and Sokoban can be formulated as MDPs\cite{Part1Jericho,Part2MCSurvey,Part1Jericho,Part1SokobanMCTS} but pose search and commonsense challenges\cite{Part1Jericho}. While A* can solve simpler puzzles, MCTS can remain effective under limited sampling and has been applied to Sokoban and Go\cite{Part1SokobanMCTS,Part1SokobanDRL,Part2GoNNMCTS}. In text games, earlier agents used neural components or hard-coded transition logic\cite{Part1Jericho,Part1NAIL}; newer approaches leverage LLM semantics and memory (e.g., MC-DML)\cite{Part2MCLLM}. Our approach similarly targets semantics-aware behavior, but shifts the LLM to an offline heuristic designer and uses logged traces to drive iterative tool improvements without per-step LLM calls\cite{Part2MCLLM}.

\textbf{Self-Evolving Agent.} Self-evolving agents learn from feedback to overcome static behavior\cite{Part3SEA} and are motivated by questions of what/when/how to evolve\cite{Part3SEA} and broader AGI aspirations\cite{Part3ToolYunjue}. Many evolve callable tools\cite{Part3ReAct,Part3ToolYunjue} or focus on autonomous tool use\cite{Part3ReAct,Part3DPAgent}; recent systems also generate/refine tools for domains such as software engineering\cite{Part3LiveSWE} and hard real-world question answering\cite{Part3ToolYunjue}. The Yunjue Agent and its in-situ self-evolving (ISE) framework emphasize task proposal and toolset optimization over interaction streams\cite{Part3ToolYunjue}. 

In contrast to runtime ``reasoning+acting'' setups (e.g., ReAct/Toolformer prompting, SWE-agent-style code execution, or LLM planning), our LLM operates offline as a synthesizer that evolves the \emph{search algorithm} itself (heuristics and phase-level control), while the MCTS executor runs without per-step LLM calls. We adapt these ideas by evolving search heuristics as reusable tools for sequential decision-making games, improving policies beyond minor action-level tweaks. For a more detailed related-work discussion and additional context, see Appendix~\ref{app:related_work}.

\section{System Architecture}
\label{sec:method}

To address the limitations of static heuristic-based planning and the high cost of training deep neural networks for state evaluation, we propose a self-evolving game agent architecture. Our system decouples the environment interaction (gameplay) from the policy improvement mechanism (LLM optimization). The architecture consists of five core components: the RL framework, the tool-evolving LLM agent, the controller, the task proposal loop, and the overall training loop.

\subsection{RL Framework}
\label{sec:method_rl}
The core gameplay engine within our architecture is a pluggable Reinforcement Learning framework. While it is adaptable to algorithms like A* or DQN, we primarily utilize a Monte Carlo Tree Search (MCTS) framework (Figure~\ref{fig:framework_overview}, right).

Crucially, the framework operates entirely offline relative to the LLM. Step-by-step LLM decision-making during gameplay is inherently slow, costly, and poorly suited for games with implicit semantics derived directly from the board status or game state. In our approach, the RL framework contains zero online LLM calls. Instead, the framework plays games at native compute speeds using programmatic heuristics, passively recording detailed gameplay traces (JSON records) that are stored offline for later optimization (as seen in the Gameplay Records box in Figure~\ref{fig:framework_overview}, right).

To make the search dynamic and adaptable, we isolate specific decision domains within the MCTS pipeline as optimizable ``heuristic slots'' (marked as blue boxes in Figure~\ref{fig:framework_overview}, right). In our experiments, we primarily optimize the \textbf{Selection} and \textbf{Simulation} phases; detailed rationale and formulations are provided in Appendix~\ref{app:mcts_slots}.

These tool functions are encapsulated as standalone Python code snippets imported by the MCTS engine at runtime. The choice of which function to load for each phase is managed by a central configuration file. Because both the Python snippets and the configuration file are plain text, the LLM can freely create, modify, or hot-swap tools iteratively after reviewing collected gameplay records.

\subsection{Tool-Evolving LLM Agent}
\label{sec:method_llm}
The LLM serves as an offline strategy improver rather than an online decision-maker. As illustrated in Figure~\ref{fig:framework_overview} (left), the policy improvement mechanism is driven by several interacting modules designed to translate raw gameplay failures into executable code improvements.

\textbf{Trace Analyzer:} The first step in the improvement pipeline is the Trace Analyzer. It acts as an LLM sub-agent that ingests raw game records within a sliding history window (e.g., the last three games by default) and extracts high-level semantics. The Analyzer is tasked with identifying \textit{why} the current strategy succeeded or failed, extracting insights on necessary rule modifications, or determining if a completely novel approach is required.

\textbf{Orchestrator and Prompt Builder:} The Orchestrator acts as the central coordinator. It receives insights from the Trace Analyzer, fetches game-specific rules and broad strategies from the Global Tools library, and retrieves the complete source code of the heuristic that produced the failure. The Orchestrator feeds this consolidated context to the Prompt Builder, which constructs the targeted prompts for code generation. By isolating the game-specific logic within Global Tools, the Orchestrator itself remains entirely game-agnostic.

\textbf{Multi-Round Query Loop:} To reliably transform abstract reasoning into executable, bug-free Python code, we utilize a multi-round query loop consisting of four distinct stages, each implemented as a single LLM round:
\begin{itemize}
    \item \textbf{Task Proposal (Analyze):} The LLM reviews the Trace Analyzer's insights against the current code to propose a concrete software modification or a new algorithmic approach.
    \item \textbf{Code Generation (Draft):} The LLM writes the initial draft of the proposed Python function.
    \item \textbf{Refinement (Critique):} The drafted code is critiqued for common logic errors, computational efficiency, and proper API usage, resulting in a refined function.
    \item \textbf{Smoke-Test Fixes:} The refined tool is compiled in a sandboxed namespace and run against a dummy environment state (a ``smoke test"). If a syntax or runtime exception occurs, the error traceback is fed back to the LLM for a single, immediate repair attempt. If the test passes, this step acts as a no-op.
\end{itemize}

\textbf{Validation and Tool Library Updates:} Following a successful smoke test, the LLM agent waits for the RL framework to execute a set of evaluation ``test-runs" using the newly generated code. The Orchestrator compares the test-run performance against the existing baseline (typically a composite score of win rate and sample efficiency). 

We only accept the newly generated code if it demonstrates a strict improvement over the current best heuristic. If accepted, the Local Tools library is populated with the new Python snippet (either as a modification of an existing tool or a brand-new creation), and the RL configuration file is updated to point the MCTS engine to this new function. This successful integration and update sequence constitutes one complete optimization loop.

\noindent\textbf{Game Environment and Optimizer:} The underlying game environment is treated as a predefined turn-based simulation. Our framework interfaces with a diverse set of games, including Sokoban, Quoridor, TextWorld, and Zork. The central ``Optimizer'' controls the RL framework's gameplay loop and orchestrates the interactions between the two primary system modules. Specifically, the Optimizer manages the flow of offline gameplay records from the RL framework to the LLM agent's Trace Analyzer, and conversely handles the dynamic online import of newly generated heuristic tool plugins from the agent back into the search engine.

\noindent\textbf{Hyperparameters:} Our architecture separates the game logic and meta-training configuration from the LLM agent's responsibilities. A dedicated hyperparameters section dictates the game type, level data, training strategy (e.g., number of test evaluation runs), and search algorithm limits (e.g., maximum MCTS rollouts). These are system-level parameters that require human pre-definition and remain strictly outside of the LLM agent's control, ensuring stable and bounded experimentation.

\noindent\textbf{Game-Agnostic Modularity:} The core competitiveness of this architecture lies in its modularity. Because the game transition logic and the hyperparameter definitions are entirely encapsulated within the top-level Controller, transferring the system to train on a completely different game requires zero changes to the underlying LLM agent or RL codebases. We simply swap the controller-level game endpoint and hyperparameter file while keeping the core modules intact.

\noindent\textbf{Framework Flexibility:} Furthermore, the architecture is not strictly tied to MCTS. Any search framework that contains optimizable, programmable components can be plugged into the right-hand ``RL Framework'' module. For instance, the static heuristic function used in the A* algorithm can perfectly serve as an optimizable ``blue box.'' By targeting the A* distance heuristic for LLM optimization instead of MCTS selection/simulation, the exact same system can evolve performant pathfinding strategies, as we will demonstrate later in Section~\ref{sec:experiments}.

\subsection{Task Proposal Loop}
\label{sec:method_task_proposal}

The task proposal loop dictates how the LLM formulates new heuristic ideas, making the initial analysis step the most critical fault point in the system. As defined by our prompt builder module, the analysis phase strictly avoids writing code. Instead, the LLM must first identify ``Key Weaknesses'' by citing specific gameplay patterns or failure traces, determine the ``Root Cause'' of these behaviors within the existing heuristic logic, and formally recommend a ``Proposed Approach.'' By design, the LLM is prompted to favor a 70/30 optimization split: approximately 70\% of the time, it should recommend an incremental modification to the existing, working code (e.g., adding a deadlock check or tweaking distance weights); the other 30\% of the time, it should propose a fundamental paradigm restructure if the current strategy has strictly plateaued. This ensures the system does not get permanently stuck in local-optima parameter tuning.

To extract consistently higher-quality proposals, future iterations could incorporate more advanced prompt engineering techniques directly into this loop. For example, replacing zero-shot analysis with rigorous Few-Shot Chain-of-Thought (CoT) reasoning, where the LLM is provided examples of successful trace-to-insight translations, could yield deeper strategic understanding. Additionally, employing a ``debate'' framework---where multiple LLM agent Personas independently critique the root cause before agreeing on a proposal---could reduce hallucinated weaknesses and lead to significantly more effective heuristic drafts.

\subsubsection{Results and Generalization}
We evaluated the learning dynamics by comparing two different training curricula: training strictly on the first 20 levels (``Curriculum A'') versus randomly sampling 20 levels from a wider pool of 50 levels per iteration (``Curriculum B''). 

Quantitative results demonstrate that the LLM-driven optimization significantly prunes the A* search space. For Curriculum A, the average node exploration dropped by \textbf{74.4\%} (from an average of 1,578 nodes in the enhanced baseline to 404 nodes in the optimized version). Curriculum B showed similar prowess, yielding a \textbf{64.8\%} average reduction across the larger level set.

More impressively, we observed localized cases of extreme improvement where the LLM identified high-impact heuristics (such as advanced deadlock detection and optimal push-distance assignment via DP). On \textbf{Level 2}, search effort was reduced from 1,223 nodes to just 4 nodes (\textbf{99.6\%} improvement). On \textbf{Level 6}, node exploration fell from 9,561 to 33 in the best iteration (\textbf{99.6\%} improvement). On \textbf{Level 17}, the search count improved from 304 to 17 (\textbf{94.4\%} improvement).

As shown in Figures~\ref{fig:sokoban_comparison_0} and \ref{fig:sokoban_comparison_2}, diversifying the training set yields findings regarding the agent's ability to resist overfitting. When the LLM optimizes a heuristic solely on Curriculum A, the generated code strongly exploits level-specific topological oddities. Conversely, the Curriculum B approach forces the prompt builder to provide the LLM with varied failure states across permutations, resulting in a more generalized code heuristic that performs more consistently across larger level distributions.

\subsubsection{MCTS Heuristic Optimization}
We run an iterative heuristic optimization process over the four standard MCTS phases (selection, expansion, simulation, and backpropagation). At each optimization iteration, we evaluate the current heuristic set on the train environments with three independent runs per game (\textit{Coin} and \textit{MapReader}). From these three runs, we compute each environment's average return and pass-rate, then rank environments by difficulty and select the three hardest failure cases for the LLM to analyze. The LLM proposes code edits for the chosen phases, warm-started from the previous iteration's heuristics. Algorithm~\ref{alg:textworld_mcts} summarizes this loop (see Appendix~\ref{app:algorithms}).

\textbf{Behavior example.}
Qualitatively, the baseline MCTS tends to over-exploit locally attractive actions (e.g., repeatedly re-entering the same rooms or oscillating between a few states), which yields deceptively stable but low-progress rollouts. After optimization, we observe more purposeful exploration: the agent covers more distinct rooms early, avoids obvious short cycles, and commits more often to trajectories that increase task-relevant signals (e.g., reaching new areas that unlock the coin or improve map coverage). A qualitative comparison of baseline vs. optimized TextWorld trajectories is provided in Appendix~\ref{app:textworld_qualitative} (Figure~\ref{fig:textworld-behavior}).

\subsubsection{Results and Generalization}
\textbf{Optimization results across MCTS phases.} Table~\ref{tab:textworld-results} reports ablations where we optimize each phase independently, as well as a full configuration. Optimizing selection improves test correctness substantially by better prioritizing promising branches, while backpropagation improves stability and reduces steps by propagating returns more effectively. Optimizing expansion alone provides limited gains and can reduce test performance, suggesting that naive widening can increase branching noise without improving long-horizon credit assignment. The strongest single-phase improvement comes from simulation, which directly improves rollout signal quality and yields the largest increase in correctness on both train and test. Combining all phase updates (full) performs near the best configuration across metrics, indicating that the phase-specific heuristics are complementary when warm-started and co-evolved. Appendix~\ref{app:textworld_combo_ablations} reports the full set of multi-phase combinations.

LLM-generated heuristics transfer more reliably to unseen environments. \section{Limitations}
\label{sec:limitations}

% - Were not able to meaningfully evaluate long-horizon behavior of tool quality
% - Open question: After ~1000 code iterations, does the tool quality collapse, plateau, or see eureka moments of continuous improvement?
% - Iterative task proposal can get stuck in parameter tuning without proposing new ideas; task proposal could use more research
% - MCTS heuristic tools show some transferability on same game with different setups
% - Lacks investigation on cross-game transferable tools
% - Evaluation limited to simple environments
% - Agent demonstrated improvements on multiple simple games, but lacked a larger, more impressive game environment to evaluate high-impact improvements

\textbf{LLM and agent scope.}
We restricted the LLM agent to read/write access only to specific tool files within a fixed RL framework (MCTS) for feasible evaluation. We did not evaluate long-horizon tool quality: after hundreds or thousands of iterations, it is unclear about heuristic quality. Expanding the agent's scope and improving how the LLM proposes and refines tools could address these limits.

\textbf{Task proposal.}
The iterative task proposal loop can get stuck in local parameter tuning without proposing structurally new ideas; when the LLM repeatedly suggests small tweaks rather than paradigm shifts, the system may fail to escape suboptimal heuristics. Task proposal quality is the most critical fault point in our pipeline. Improving the diversity and effectiveness of proposals is a central direction for future work.

\textbf{Generalization.}
We observed same-game differnt level transfer but did not investigate cross-game transfer. Evaluation was limited to relatively simple or medium-complexity, white-box environments with explicit code APIs. We lacked both a larger, more demanding game benchmark and any setting where the agent must interact with black-box games (e.g., via pixels or screenshots).

\section{Future Work}
\label{sec:future}

\textbf{Broader agent scope.}
We limited the LLM agent's read/write access to specific tool files within a fixed RL framework (MCTS) to keep evaluation feasible and controlled. The agent might perform better with elevated scope: access to full tool folders, the freedom to switch or combine RL frameworks that interact with the environment, or even the ability to modify its own prompt engine and LLM workflow. Exploring such expanded agent boundaries could reveal whether self-evolving tool creation scales with greater autonomy.

\textbf{Task proposal effectiveness.}
Improving the quality and diversity of task proposals is important. Techniques such as few-shot chain-of-thought reasoning, multi-agent debate over root causes, or explicit curriculum over failure modes could help the LLM propose more impactful and varied heuristic changes, reducing the risk of getting stuck in parameter tuning.

\textbf{Generalizing beyond white-box games.}
The game environments we evaluated are white-box: they expose explicit code APIs for state queries and actions. A natural extension is to allow the LLM agent to discover or implement interaction frameworks for black-box games---for example, using screenshots or pixel observations to learn the game interface and then implementing tools that produce encapsulated action sequences (e.g., mouse clicks or key presses). Such a step would broaden the applicability of the self-evolving tool paradigm to games without programmatic access.

\subsection{Detailed Related Work}
\label{app:related_work}
\textbf{Monte Carlo Tree Search.}
Monte Carlo Tree Search (MCTS) method excels at deriving optimal decisions in environments with complex decision space because it balances efficient exploration through monte carlo sampling and exploitation of promising subparts of the search tree\cite{Part2MCSurvey} \cite{Part2MCTSCoulom}. Over time, MCTS demonstrates robust applications in real-world problems like combinatorial games\cite{Part2MCSurvey}\cite{Part2GoNNMCTS} and policy optimization \cite{Part2PGTS}, along with immense potential in its improvements. Other than the theoretical breakthroughs in MCTS, such as better exploration heuristics like PUCT \cite{Part2MCSurvey}, studies of MCTS also utilized the incorporations with rapidly developing machine learning techniques like neural networks and LLM \cite{Part2MCLLM}\cite{Part2GoNNMCTS}. In AlphaGo, MCTS searches through the game state search tree to guide better policies, while neural network's capability to provide accurate predictions of state values for MCTS\cite{Part2GoNNMCTS}. With the progression of studies in large language models, MCTS thrives in empowering novel LLM agents through various forms of applications, ranging from applying MCTS as a tool for improving LLM performances \cite{Part2PGTS} to incorporating LLM as a component of the MCTS\cite{Part2MCLLM}. Modern incorporations between MCTS and LLM demonstrates more utilization of LLM's reasoning capability and autonomy, such as applying LLM as a heuristic optimizer for MCTS\cite{Part2PlanHeuris}. Our work recognizes the foundational strength of MCTS in deriving solutions in complex decision space, such as complex combinatorial games\cite{Part2GoNNMCTS}, and the forefront research products in incorporating state-of-the-art machine learning techniques into the MCTS framework, such as optimizing MCTS components with LLM\cite{Part2PlanHeuris}. In such, our work attempts to adapt a MCTS framework while optimizing it with LLM agents. 


\textbf{Sequential Decision-Making Games.}
Sequential decision-making games are defined as games that requires the iteration of observing the state, issuing an action, and receiving the state change and potential rewards\cite{Part1Jericho}\cite{Part2MCSurvey}. Solve such game environments with algorithms, like interactive-fiction (IF) games like Zork\cite{Part1Jericho} and turn-based games like Go and Sokoban\cite{Part1SokobanMCTS}, enjoys benefit in their conversion into Markov decision processes\cite{Part1Jericho} while addresses challenges like the explosion of combinatorial state or action spaces, the requirement for commonsense reasoning, and the demand for effective knowledge representation\cite{Part1Jericho}. In such, though shortest path-finding algorithms like A* presents perfect solutions for simple puzzle games, they might fall short in complex combinatorial games due to state space explosion. MCTS, due to its capability of exploiting optimal solution with limited sampling, emerged as a choice to solve complex games like complex Sokoban maps\cite{Part1SokobanMCTS}\cite{Part1SokobanDRL} or Go, leading to breakthroughs like the high performance of AlphaGo \cite{Part2GoNNMCTS}. However, early explorations in solving test-based games with MCTS mainly focused on incorporation with neural network\cite{Part1Jericho} or hardcoding transition probabilities for specific states and actions \cite{Part1NAIL}. Such methodologies undermines the rich semantic meanings in state descriptions and might be less transferrable across different environments. The emerge of better large language models led to work that utilized LLM to optimize MCTS heuristics, rather than using neural network outputs or hardcoded instructions. MC-DML \cite{Part2MCLLM} attempts to capture the semantic meaning and past game experiences by deploying LLM agent as the initial policy of the MCTS framework, while using a dynamic memory mechanism to efficiently preserve the knowledge base of in or cross-game experiences. Building on the recent incorporation of LLM into MCTS for solving games, our work attempts to optimize different components of MCTS framework with LLM agent, in order to utilize LLM's strength in reasoning to capture the semantic meanings and higher-level logic in our target sequential-decision making games. Compared to past work like MC-DML \cite{Part2MCLLM}, our work attempts to save compute cost by adapting LLM as heuristic designer, rather than stepwise policy maker. We also address the challenge of knowledge base management by organizing and using past game traces to drive better heuristics with LLM.

\textbf{Self-Evolving Agent.}
Unlike static LLMs, which demonstrate less capability to adapting to novel or evolving contexts, self-evolving agents are designed to overcome such limitation by learning from real-world feedbacks and optimize its tasks\cite{Part3SEA}. The emergence of self-evolving agents attempts to redefine the AI frontier closer towards AGI\cite{Part3ToolYunjue}, while inspiring the grounding of the self-evolving agent concepts into appropriate domains that can address the questions of \textit{what to evolve, when to evolve, and how to evolve}\cite{Part3SEA}. Many recent works applies the concept of self-evolving agent with agent tools, which are callable components like code scripts that assist the agent to complete certain tasks \cite{Part3ReAct}\cite{Part3ToolYunjue}, as a target to evolve. Some past work have focused more on the agent's capability of using existing external tools\cite{Part3ReAct}, leading towards agents that can solve tasks with more autonomy \cite{Part3DPAgent}. The emerging research of self-evolving agent further enhanced such strength in higher agent autonomy, as more recent works about self-evolving agents explore the agent's workflow to generate and optimize the tools under a specified domain, such as software engineering \cite{Part3LiveSWE} and challenging real-world question solving \cite{Part3ToolYunjue}. In particular, the Yunjue Agent solves a real-world question prompt through task proposal, which the agent proposes a stream of subtasks and produce tools to gradually solve the entire problem\cite{Part3ToolYunjue}. The Yunjue Agent further highlights its strength in self-evolving concept by proposing the In-situ self evolving (ISE) framework that treats a group of task interactions as a stream for continuous knowledge accumulation and toolset optimization\cite{Part3ToolYunjue}, and this framework enables the agent to familiarize and solve diverse real-world question prompts with less human supervision. Our work built on the emerging research of self-evolving agent and agent tool generation, as we attempt to tune the LLM agent to continuously produce and optimizer MCTS heuristics as tools to solve sequential decision-making games with less human supervision. In particular, our work demonstrates an application of self-evolving agent in the domain of complex, sequential decision-making games. 


\subsection{TextWorld evaluation setup}
\label{app:textworld_setup}
We evaluate three environment variants (deterministic, stochastic, and punishment) and average results over three random seeds (0--2). For each optimization iteration, training uses a disjoint set of seeds (3--5). We train and test on both Coin and MapReader using a level-size curriculum: training instances use 2/4/6/8 locations, while testing uses harder 5/7/9/11/13-location instances. For Coin, all instances include doors and use zero distractor items. For MapReader, we disable doors and use zero distractor items per location, with the maximum distance apart increasing with the number of locations (1--4 for training; 3--7 for testing). All MCTS phase ablations are drawn from the same four phases: selection, expansion, simulation, and backpropagation.

\noindent\textbf{Notes.} This setup cleanly separates \emph{within-configuration robustness} (across stochasticity/punishment variants and seeds) from \emph{generalization} (to larger, harder held-out maps). The disjoint seed split also prevents leakage of memorized trajectories between optimization iterations and evaluation, so improvements more directly reflect the quality of the evolved heuristics rather than lucky rollouts.

\subsection{MCTS heuristic slots (Selection and Simulation)}
\label{app:mcts_slots}
\noindent\textbf{Selection.} In traditional MCTS, the best child is chosen using a static heuristic such as Upper Confidence bounds applied to Trees (UCT). We encapsulate this selection logic as a programmable tool function, allowing the LLM to write environment-specific rules for exploring the search tree.

\noindent\textbf{Simulation.} Standard simulation rollouts often fall back on iteratively selecting uniformly random legal actions until a terminal state is reached. We reinterpret the rollout policy as a heuristic function in itself. An LLM-designed simulation tool can introduce targeted exploration or score-guided action selection to improve sample efficiency without requiring deeper searches.

\subsection{Algorithms}
\label{app:algorithms}
\noindent This section collects the core algorithmic loops referenced in the main text for completeness.

\noindent\textbf{Takeaway.} Across both Sokoban (A*) and TextWorld (MCTS), the implementation is intentionally structured to expose a small number of \emph{pluggable} decision points to the LLM (e.g., a heuristic function in A*, or phase-specific heuristics in MCTS). This keeps the planner mechanics fixed while allowing the tool-evolution loop to make targeted, auditable code changes.

\noindent Algorithm~\ref{alg:sokoban_astar} shows the A* search skeleton used in our Sokoban experiments. The key design point is the pluggable heuristic call (highlighted in bold), which is the only component modified by the LLM across optimization iterations.

\noindent Algorithm~\ref{alg:textworld_mcts} summarizes the TextWorld optimization loop: repeated train rollouts are logged, the hardest failure cases are selected, and the LLM proposes phase-specific heuristic updates that are then evaluated on held-out test environments.


\begin{algorithm}[htbp]
\caption{Sokoban A* Search with Pluggable Heuristic}
\label{alg:sokoban_astar}
\begin{algorithmic}[1]
\STATE $open\_heap \gets \{ \text{initial\_state} \}$
\STATE $closed\_set \gets \emptyset$
\WHILE{$open\_heap \neq \emptyset$}
    \STATE $current \gets \text{HeapPop}(open\_heap)$
    \IF{$current$ meets goal conditions}
        \RETURN solution
    \ENDIF
    \IF{$current.hash\_key \in closed\_set$}
        \STATE \textbf{continue}
    \ENDIF
    \STATE $closed\_set \gets closed\_set \cup \{current.hash\_key\}$
    \FOR{each valid move generating $new\_state$}
        \STATE $new\_g \gets current.g + 1$
        \STATE $\mathbf{new\_h \gets heuristic(board, new\_state.boxes)}$ \hfill $\leftarrow$ \textbf{Optimized by LLM}
        \STATE $new\_state.f \gets new\_g + \mathbf{new\_h}$
        \STATE $\text{HeapPush}(open\_heap, new\_state)$
    \ENDFOR
\ENDWHILE
\RETURN failure
\end{algorithmic}
\end{algorithm}

\begin{algorithm}[htbp]
\caption{TextWorld MCTS with Iterative LLM Heuristic Optimization}
\label{alg:textworld_mcts}
\begin{algorithmic}[1]
\STATE Choose phases $\mathcal{P} \subseteq \{\textsc{Sel},\textsc{Exp},\textsc{Sim},\textsc{Back}\}$
\STATE $h^{(0)} \gets$ baseline heuristics
\FOR{$t \gets 1$ to $N$}
  \STATE $h^{(t)} \gets h^{(t-1)}$
  \STATE $\mathcal{D} \gets \emptyset$
  \FOR{$g \in \{\textit{Coin},\textit{MapReader}\}$}
    \FOR{$r \gets 1$ to $3$}
      \STATE $\mathcal{D} \gets \mathcal{D} \cup \text{RunTrain}(g,\; \text{MCTS}(h^{(t)}))$
    \ENDFOR
  \ENDFOR
  \STATE Compute $(\overline{R}_e,\; \text{Pass}_e)$ from the 3 runs for each train env $e$
  \STATE $\mathcal{F} \gets \text{TopKHard}(\mathcal{D},\; K{=}3,\; \text{key}{=}(\overline{R},\text{Pass}))$
  \STATE $h^{(t)}_{\mathcal{P}} \gets \text{LLMOptimize}(h^{(t)}_{\mathcal{P}},\; \mathcal{F})$
\ENDFOR
\STATE Evaluate \text{MCTS}($h^{(N)}$) on unseen test environments
\end{algorithmic}
\end{algorithm}

\subsection{Comparison with baseline on Zork}
